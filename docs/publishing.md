# Publishing

`publish.bash` is the single entry point. It picks drafts, validates them, lets you review link suggestions, writes the published files, and regenerates the index.

```bash
./publish.bash               # Interactive picker
./publish.bash --all         # Publish every draft
./publish.bash <file>.md     # Publish a single file by name or relative path
```

Files passed to the third form are matched against the relative path under `drafts/` and against the bare filename. Both `./publish.bash welcome.md` and `./publish.bash Internal/welcome.md` work.

## What it does, per file

```
1. Verify authors        →  prompts to register unknown usernames
2. Validate assets       →  abort if any assets/* reference is missing
3. Detect local links    →  scan for mentions of existing wiki page titles
4. Search Wikipedia      →  scan body for terms with Wikipedia articles
5. Apply selected links  →  user opts into each via JSON file
6. Tag derivation        →  add domain/topic/subtopic from directory path
7. Write wiki/<slug>.md  →  serialize processed frontmatter + body
8. Regenerate index      →  rebuild wiki/_index.json (+ chunks if needed)
```

If a step fails (missing asset, declined author registration), that file is skipped and the next selected draft is processed. The batch never bails out for a single failure.

## Author registration

Before the link-detection step, the helper checks every `author:` and `co-authors:` username against `authors.json`. Unknown ones prompt:

```
Warning: author 'jane' is not registered.
Create author 'jane'? [Y/n] y
Full name for 'jane': Jane Doe
Created: jane → Jane Doe
```

Decline (`n`) or provide an empty name → that file is aborted. Accept → the new entry is written to `authors.json` and the publish continues. See [authors.md](authors.md).

## Asset validation

A regex over the draft body collects every `(![alt](url)` and `[text](url))` whose `url` starts with `assets/`. Each path must resolve under `assets/`. See [assets.md](assets.md).

## Link suggestions

Two parallel scans run for each file and produce two JSON files in `.publish/`:

### Local wiki links

`.publish/local-links.json` lists titles of existing wiki pages mentioned in the draft body, with occurrence counts and a context snippet:

```json
{
  "_info": "Set \"include\" to true for local wiki links you want to add.",
  "_file": "machine-learning.md",
  "links": [
    { "include": false, "title": "Linear Algebra", "slug": "linear-algebra", "occurrences": 3, "context": "...vectors and matrices in [Linear Algebra]..." }
  ]
}
```

Edit the file in your editor, set `"include": true` for the suggestions you want, save, and press Enter in the terminal. Each accepted suggestion converts the **first occurrence** of the title into `[Title](slug)` and adds `slug` to the page's `related` field. Subsequent occurrences are left as plain text (Wikipedia-style).

### Wikipedia links

`.publish/wikipedia-links.json` is generated by [scripts/wikipedia-search.sh](../scripts/wikipedia-search.sh). It searches a curated pattern list ([scripts/patterns.txt](../scripts/patterns.txt)) and the body for terms with Wikipedia articles. The format is the same as local-links.json. Accepted entries are converted to `[Term](https://en.wikipedia.org/wiki/Term)` (rendered with a "W" indicator in the UI).

If a draft has no link suggestions, the prompt is skipped automatically.

## Helper subcommands

`publish.bash` is a thin shell wrapper over [project/scripts/publish-helper.js](../project/scripts/publish-helper.js). The helper exposes these subcommands:

```
node project/scripts/publish-helper.js <command> [args]

generate-local-links <draft>   # build .publish/local-links.json
check-authors <draft>          # print MISSING:<username> for each unknown author
create-author <username> <name>  # add to authors.json
publish <draft>                # apply links + write wiki/<slug>.md
```

Useful for debugging or scripting non-interactive flows.

## Re-publishing

Re-running `./publish.bash` on a file that already exists in `wiki/` is fine — it overwrites. The publish status in the interactive picker shows `UPDATE` for existing slugs and `NEW` otherwise.

## Index regeneration

The last step in every publish run is `node project/scripts/generate-index.js`. It:

- Reads every `wiki/*.md` (skipping files starting with `_`).
- Builds taxonomy from the `drafts/` directory tree.
- Embeds the contents of `authors.json`.
- Splits pages into 10 MB chunks (see [architecture.md](architecture.md)).
- Writes the base + any rotated chunk files.
- Deletes stale chunk files from previous runs.
